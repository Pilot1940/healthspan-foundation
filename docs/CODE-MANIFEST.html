Non-Negotiable Rules

1
Never hardcode thresholds — all live in system_config. Read via lib.contract.get_config(conn, key) (Python) or system_config SELECT (Edge fn).
2
AI extraction output goes to stg_*_review staging first, never directly to prod tables.
3
Every new table needs an RLS policy in the same migration. DELETE/TRUNCATE revoked from authenticated + healthspan_app; ALL revoked from anon.
4
Test files mirror source structure: src/foo.py → tests/unit/test_foo.py.
5
Commit format: feat: | fix: | schema: | test: | docs: | chore: | adr:
6
Never modify auth logic without explicit instruction.
7
When stuck for more than 2 attempts, stop and ask.

telegram-webhook · whoop-webhook · whoop-oauth

Supabase Storage

Private health-media bucket. Telegram photos uploaded here by telegram-webhook; inbox_drain fetches via signed URLs (300s TTL). No health data in any URL param.

Claude Code Routine (API-triggered)

Cloud-hosted, laptop-independent. telegram-webhook fires an HTTP POST to ROUTINE_TRIGGER_URL (Supabase secret). Routine runs python -m monitor.inbox_drain --once and reports JSON summary.

Auth: healthspan.drainer@chitalkar.com · reads HS_AUTH_EMAIL, HS_AUTH_PASSWORD, HS_ANTHROPIC_API_KEY, TELEGRAM_BOT_TOKEN

WHOOP API

Webhooks push {type, id, user_id} to whoop-webhook. Routine syncs: workouts, cycles, recovery, sleep.

OAuth tokens in whoop_tokens. Lazy refresh-on-use. HMAC-SHA256 signature verified on every event.

Telegram Bot

Inbound: telegram-webhook receives messages → enqueues to media_inbox → fires Routine. Outbound: whoop-webhook and Routine send push notifications via sendMessage.

Auth: X-Telegram-Bot-Api-Secret-Token header · chat_id → profile_id via telegram_identities

Anthropic Claude API

Used by inbox_drain --once for vision extraction (food/lab/supplement photos), content-clustering ungrouped items, and (via monitor/brief.py) generating rest-of-day action suggestions. Model configurable via system_config.drain.vision_model; brief model via system_config.brief.model.

Env: HS_ANTHROPIC_API_KEY (platform reserves the plain name)

AWS S3 + SES

S3: media backup. SES: admin alerts (separate from Telegram push path).

PY monitor/brief.py v3.13.0 — new

Daily health brief composer. Called from run_once() after every adult food write; also runnable directly via CLI. Assembles a structured Telegram message from four data sources, then asks Claude for 2-4 concrete rest-of-day actions. Minor-safe: Viome section suppressed; growth/performance framing in food and actions.

  • compose_brief(db, profile_id, cfg, api_key, token, today) → str — looks up chat_id + is_minor from telegram_identities; loads targets from lib.context.load_context.
  • Food section: _fetch_food_full — kcal, protein, carbs, fat, meal count from food_logs. Shown vs targets with remaining. Minor: growth-fuelling nudge if <60% of target.
  • Supplements section: _fetch_supplement_status — active regimen entries (name + timing[]) + today's supplement_intake_logs. Groups by timing slot (morning/lunch/dinner/bedtime/anytime); ✅ taken / ⬜ not taken; current slot marked ←.
  • WHOOP section: _fetch_whoop — most recent whoop_cycles row (recovery%, HRV ms, RHR bpm, sleep h:mm). Stale flag if cycle_start < today.
  • Viome section (adult only): _fetch_today_viome_flags — today's food_logs rows where verdict IS NOT NULL AND verdict <> 'clean'; surfaces flags[] for avoid/minimize + superfoods.
  • Actions section: _call_claude_actions — calls Claude with a structured context prompt; returns 2-4 numbered action sentences. Model: system_config.brief.model (default: claude-haiku-4-5-20251001). Best-effort — empty string on failure.
  • Sends via telegram_send (lazy import from inbox_drain — avoids circular import at module level).
  • CLI: python -m monitor.brief --profile-id <uuid> [--today YYYY-MM-DD]
